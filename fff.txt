# Сравнение лабораторных работ по Terraform (0-6)

## Общий рабочий цикл (Workflow)
Для любой лабораторной работы (кроме специфических команд) используется стандартный цикл:
1. cd labX — переход в папку лабораторной.
2. terraform init — инициализация (загрузка провайдеров).
3. terraform plan — просмотр планируемых изменений.
4. terraform apply -auto-approve — применение конфигурации.
5. terraform destroy -auto-approve — удаление всех созданных ресурсов.

## Лабораторная работа 0: Подготовка (Git + Ansible + Docker)
*Это предварительный этап, описанный в instructions.txt.*
*   Цель: Настройка окружения.

## Лабораторная работа 1: Docker + Jenkins
*   Цель: Управление локальными (контейнерными) ресурсами.
*   Основные инструменты: Провайдер docker.
*   Что нового:
    *   Создание Docker-образов (docker_image).
    *   Запуск контейнеров (docker_container).
    *   Настройка портов и переменных окружения для Jenkins.
*   Отличие: Работа происходит не в облаке, а в локальном Docker-окружении.

---

## Лабораторная работа 2: Базовый инстанс AWS
*   Цель: Знакомство с облачным провайдером AWS.
*   Основные инструменты: Провайдер aws, ресурс aws_instance.
*   Что нового:
    *   Настройка региона (region).
    *   Запуск простейшей виртуальной машины (EC2) на Ubuntu.
*   Отличие: Первый выход в облако, использование AMI (Amazon Machine Image).

---

## Лабораторная работа 3: Переменные (Variables), Locals и Outputs
*   Цель: Сделать конфигурацию гибкой и переиспользуемой.
*   Что нового:
    *   variables.tf — описание входных параметров.
    *   .tfvars файлы — разделение настроек для разных сред (dev/prod).
    *   locals — внутренние переменные для вычислений/упрощения кода.
    *   output — вывод данных (например, IP-адреса) после завершения работы.
*   Отличие: Уход от "хардкода" (жестко прописанных значений).

---

## Лабораторная работа 4: IAM (Users, Groups, Policies)
*   Цель: Управление доступом в AWS.
*   Что нового:
    *   Создание пользователей (aws_iam_user) с использованием цикла count.
    *   Создание групп (aws_iam_group) и политик доступа (aws_iam_policy).
    *   Использование data "aws_iam_policy_document" для описания прав в формате JSON.
*   Отличие: Работа не с инфраструктурой (серверами), а с правами доступа и пользователями.

---

## Лабораторная работа 5: Сетевая инфраструктура (VPC) и Dynamic Blocks
*   Цель: Создание полноценной защищенной сети.
*   Что нового:
    *   Создание сети (VPC), подсетей (Subnets) и интернет-шлюза (IGW).
    *   Dynamic Blocks: Автоматическое создание правил безопасности (Security Groups) через циклы for_each внутри ресурса.
*   Отличие: Глубокая настройка сети и использование продвинутых конструкций Terraform для сокращения кода.

---

## Лабораторная работа 6: Data Sources
*   Цель: Получение данных о существующих ресурсах.
*   Что нового:
    *   Блок data — позволяет "прочитать" информацию о ресурсе, который уже создан или управляется не только этим скриптом.
*   Отличие: Если resource создает что-то новое, то data только запрашивает информацию.

---

## Как это сделать в целом (Рекомендации)
1.  Постепенность: Начинайте с Lab 1, чтобы понять логику Terraform, затем переходите к AWS (Lab 2).
2.  Структура: Всегда разделяйте файлы (main.tf, variables.tf, outputs.tf), когда проект становится большим (как в Lab 3-5).
3.  Безопасность: Никогда не храните ключи доступа AWS в коде. Используйте aws configure.
4.  Автоматизация: Используйте locals и dynamic blocks (как в Lab 5), чтобы не повторять один и тот же код многократно.
5.  Уборка: Всегда делайте terraform destroy, особенно после работы с AWS, чтобы избежать лишних трат.
